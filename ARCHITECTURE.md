# PDF Voice — Целевая архитектура

Документ описывает **целевую** структуру движка «PDF → текст → речь»: как сделать
её робастной ко всем типам PDF (проза, сканы, учебники/статьи, смешанные файлы) и
дешёвой к расширению (новые языки, новые голоса/движки) — **реализуя пока только
русский**.

Это план, не отчёт о сделанном. Текущий код (`TextNormalizer`, `SpeechEngine`,
`PDFTextExtractor`/`OCRTextExtractor`) — рабочий, но монолитен по двум осям: язык
вшит в нормализацию, а движок Silero вшит в `SpeechEngine` через `if`. План убирает
именно эти два шва.

## Версия-предпосылка
iOS ≥ 16.0, Swift ≥ 5.9. Все `async`/`@MainActor`/`AsyncStream` ниже действительны
при этом. Если базовая версия другая — пересмотреть сигнатуры бэкендов.

---

## 1. Принципы (чем руководствуемся)

1. **Разделять язык-зависимое и язык-независимое.** Разбиение на строки, карта
   смещений, склейка переносов, детект номеров страниц, cross-page merge — механика,
   общая для всех языков. Аббревиатуры, числа-словами, ударения, слова-разделы —
   русское. Их нельзя держать в одном `enum`.
2. **Решение text/OCR — постранично, не на весь документ.** Это разом закрывает
   смешанные PDF и убирает асимметрию «OCR читается хуже текста» (оба пути идут через
   один конвейер очистки).
3. **Движок синтеза — за швом.** Конкретный синтез (AVSpeech / Silero / облако) —
   за протоколом `SpeechBackend`, который владеет **порядком проигрывания**.
   `SpeechCoordinator` владеет **навигацией, подсветкой, Now-Playing, прерываниями**
   (точное разделение — §3.4).
4. **YAGNI-граница.** Сейчас строим только *швы* (границы), за которыми живёт
   **единственная** русская реализация. Полноценный мультиязычный фреймворк (RTL,
   локали, переговоры о языке) — НЕ строим, пока нет второго языка с реальными
   требованиями.

## 2. Конвейер (целевой)

```
PDF-файл
  │
  ▼
classifyPage(_:) ─► решает для КАЖДОЙ страницы: .text | .ocr | .skip   (функция, не протокол)
  │                 .skip — пустые/декоративные страницы, без Vision-прохода
  ▼
PageTextSource ──► TextLayerSource (PDFPage.string) | OCRSource (Vision)
  │                 → RawPage { lines | observations, boxes, offsets }
  │                 ⚠ OCR-страницы идут отдельным медленным lane (см. §3.6)
  ▼
TextPipeline ─────► (НА КАЖДОЙ СТРАНИЦЕ) split строк, offset-карта, склейка переносов,
  │                 boilerplate/номера страниц/leader-строки   (ЯЗЫК-НЕЗАВИСИМО)
  ▼
orderReading(_:) ─► порядок чтения (сейчас identity; позже — колонки/таблицы/код)
  │
  ▼
LanguageProfile ──► токенизация (локаль) + isHeading   ← RU only
  │                 → [Sentence] (СЫРОЕ очищенное предложение + offsets + language;
  │                   range=nil для OCR-страниц, подсветка через boxes)
  ▼
TextPipeline.mergeCrossPage([Sentence]) ─► склейка предложений, разрезанных границей
  │                 страницы. ПОСЛЕ токенизации — слить можно только когда предложения
  │                 уже существуют (как в текущем PDFTextExtractor).
  ▼  ── кэш пишется здесь: сырое предложение, не результат синтеза ──
  ▼
SpeechCoordinator ► навигация (play/skip/speed), currentIndex, подсветка из событий,
  │                 Now-Playing, прерывания
  ▼
SpeechBackend ────► РЕНДЕРИТ разметку в свой формат (U+0301 / «+» / SSML) и синтезирует
                    AVSpeechBackend | SileroBackend | (будущее) CloudBackend
```

Ключевой принцип потока: **раскрытие для речи (`expandForSpeech`) применяется поздно —
на воспроизведении, а не при извлечении.** Кэш хранит сырое очищенное предложение,
поэтому улучшение лингвистики (Фаза 1) не инвалидирует кэш, а движок-специфичная
разметка ударений рендерится каждым `SpeechBackend` по-своему.

Общая механика конвейера живёт в `TextPipeline` и НЕ дублируется между text- и
OCR-путём (сейчас дублируется). Очистка строк/boilerplate/offset — **per-page, до**
токенизации; `mergeCrossPage` — **отдельная стадия после** токенизации (поверх всего
`[Sentence]`). Boilerplate-детект частотный и язык-независимый → он в `TextPipeline`,
а НЕ в `LanguageProfile`.

## 3. Швы (границы) — то, что строим сейчас

Часть швов — протоколы (две+ реализации уже есть), часть — обычные функции (одна
реализация; протокол был бы спекуляцией). Это помечено в каждом подразделе.

### 3.1 Постраничный детект — функция, не протокол
Заменяет документ-уровневый `hasTextLayer` (первые 5 стр → всё text/OCR).
Реализаций ровно одна (плотность букв), поэтому это **функция**, а не протокол —
протокол извлечём, только если появится вторая стратегия классификации.

```swift
enum PageKind {
    case text          // годный текстовый слой (плотность букв ≥ порога)
    case ocr           // битый/отсутствующий слой → Vision
    case skip          // пусто/декор (< N значимых символов) → не тратим Vision
}

func classifyPage(_ page: PDFPage) -> PageKind
```

`.skip` важен: без него пустые страницы-разделители глав уходили бы в бессмысленный
OCR-проход. Документ-уровневый быстрый путь остаётся как оптимизация: если выборка
страниц вся `.text` — не гоняем классификатор на каждой.

Это решает **смешанные PDF** и делает OCR/текст единым потоком.

### 3.2 `PageTextSource` — откуда берётся текст
```swift
struct RawPageLine { let text: String; let startUTF16: Int; let box: CGRect? }
struct RawPage { let pageIndex: Int; let lines: [RawPageLine] }

protocol PageTextSource {
    func read(page: PDFPage, at index: Int) async -> RawPage
}
// TextLayerSource — синхронно из PDFPage.string (box = nil, range через offsets)
// OCRSource      — Vision, box заполнен, offsets синтетические
```

После этого **OCR и текст идут через ОДНУ очистку** → колонтитулы, номера страниц,
паузы заголовков, cross-page merge работают и на сканах (сейчас не работают).

### 3.3 `LanguageProfile` — весь язык в одном месте
```swift
protocol LanguageProfile {
    var code: String { get }                         // "ru"

    /// Токенизация предложений (NLTokenizer с правильной локалью).
    func sentences(in cleaned: String) -> [Range<String.Index>]

    /// Похоже ли на заголовок (для пауз и не-склейки).
    func isHeading(_ raw: String) -> Bool

    /// Раскрытие для синтеза. Возвращает НЕ плоскую строку, а РАЗМЕЧЕННОЕ
    /// представление: текст + позиции ударений + паузы. Движок-специфичный рендер
    /// (U+0301 / «+» / SSML) делает SpeechBackend, не профиль (см. §3.4).
    /// Применяется ПОЗДНО — на воспроизведении, не при извлечении (см. §4).
    func render(_ sentence: String) -> SpokenMarkup
}

struct SpokenMarkup {
    let text: String                 // раскрытые числа/аббревиатуры
    let stresses: [Int]              // индексы ударных гласных (UTF-16)
    // Паузы — уровня предложения (pauseBetweenSentences + headingPause), ими владеет
    // координатор. Позиционные паузы внутри предложения НЕ добавляем, пока их не
    // потребует фича (YAGNI).
}
```

`RussianProfile` композирует существующую логику, разнесённую по файлам:
- `RussianNumerals`     — числа→слова (+ будущее согласование по падежу/роду)
- `RussianAbbreviations`— словарь + разрешение «г.»/«в.»/«см» по контексту
- `RussianStress`       — омографы: возвращает ПОЗИЦИИ ударений, не вставляет символ
- `RussianHeadings`     — «Глава N», спец-разделы (пролог/эпилог/…)

**Важно — что НЕ в профиле:** язык-независимая механика (split строк, offset-карта,
склейка переносов, `isLeaderLine`, `isPageNumberLine`, **boilerplate-детект**,
`mergeCrossPage`) живёт в `TextPipeline`. Boilerplate частотный → язык-независим.
Профиль — только то, что меняется от языка к языку.

### 3.4 `SpeechBackend` — движок синтеза за швом
Убирает `if sileroServerURL != nil` из `SpeechEngine`.

```swift
protocol SpeechBackend: AnyObject {
    /// Проиграть последовательность с index. Бэкенд САМ владеет порядком
    /// проигрывания (AVSpeech enqueue-all; Silero fetch-play-loop с предзагрузкой) —
    /// это сохраняет фикс «очередь сразу, без повторного чтения». Координатор НЕ
    /// микроменеджит очередь, он лишь задаёт намерение и читает события.
    /// Бэкенд РЕНДЕРИТ SpokenMarkup в свой формат ударений (U+0301 / «+» / SSML).
    func play(_ sentences: [Sentence], from index: Int,
              speed: Double, voice: VoiceSelection,
              render: @escaping (Sentence) -> SpokenMarkup) -> AsyncStream<SpeechEvent>
    func pause(); func resume(); func stop()
    // Смена скорости/голоса на лету = stop + play(from: current) — для AVSpeech это
    // штатное пере-наполнение очереди, как в текущем коде.
}

enum SpeechEvent { case didStart(Int), didWord(Range<String.Index>), didFinish(Int), ended }
```

**Разделение ответственности (исправляет неоднозначность раннего черновика):**
- `SpeechBackend` владеет **порядком проигрывания и рендером разметки**.
- `SpeechCoordinator` (рефакторинг `SpeechEngine`) владеет **навигационным намерением**
  (play/pause/skip/seek/speed), **состоянием UI** (подсветка из `.didStart`/`.didWord`),
  `NowPlayingController` и обработкой прерываний; подписан на `AsyncStream` бэкенда.

AVSpeech (enqueue-all + delegate) и Silero (fetch-play-loop) прячутся за одинаковым
стримом событий. Координатор очередь не дублирует.

### 3.5 Порядок чтения — функция, не протокол
```swift
func orderReading(_ page: RawPage) -> RawPage   // сейчас: identity (return page)
// Позже:  кластеризация боксов по X (колонки в статьях/учебниках),
//         пропуск/замена листингов кода и формул на «листинг».
```

Нужно под ваш корпус (учебники/статьи), реализуется в Фазе 2. Сейчас — обычная
функция-заглушка `return page`, а НЕ протокол: вторая реализация ещё не существует,
протокол был бы спекуляцией. Извлечём протокол, когда появится реальный второй
вариант упорядочивания.

### 3.6 Загрузчик: отдельный lane для OCR (латентность гетерогенна)
Текстовая страница — миллисекунды, OCR — 1–2 секунды. Наивное слияние в один батч
(текущие батчи по 50) подвесит батч, если в нём окажется скан. Поэтому при смешанном
документе:
- `.text`-страницы грузятся быстрым потоком как сейчас (прогрессивно, батчами);
- `.ocr`-страницы уходят в **отдельную низкоприоритетную очередь** и дописываются в
  `speech.appendSentences` по мере готовности;
- `loadedPageCount`/кэш отмечают страницу готовой только когда её источник отработал
  (для скана — после OCR).

Это сохраняет «мгновенный старт» на текстовых страницах и не даёт сканам блокировать
загрузку остального текста.

## 4. Модель данных и кэш — храним сырое, рендерим поздно

**Ключевое исправление архитектуры.** Сейчас `Sentence.text` = уже озвученный текст
(`expandForSpeech` применён при извлечении), и именно он лежит в кэше. Из-за этого
любое улучшение лингвистики (Фаза 1: ударения, числительные) делает все кэши
устаревшими → пере-OCR книг на десятки МБ. Это прямо мешает «дёшево добавлять фичи».

**Решение:** `Sentence` хранит **сырое очищенное предложение** (для подсветки и
токенизации); `render(_:)` → `SpokenMarkup` применяется на воспроизведении (он
дешёвый). Тогда улучшение лингвистики кэш НЕ трогает, а движок-специфичная разметка
ударений считается под конкретный `SpeechBackend`.

```swift
struct Sentence {
    let rawText: String    // СЫРОЕ очищенное предложение (не озвученное)
    let pageIndex: Int
    let range: NSRange?
    let boxes: [CGRect]
    let isHeading: Bool
    let language: String   // "ru" — задел под мультиязычность
}

struct EncodedSentence: Codable {
    // ...существующие поля; text → rawText...
    let language: String?  // старые кэши → nil → "ru"
}
```

`Sentence` получает язык; пишем всегда `"ru"`. Добавление поля `language` обратно
совместимо (старые кэши → nil → "ru", приём `isHeading: Bool?`).

⚠️ **Но сам переход `text → rawText` — одноразовая миграция, а не zero-cost.** Старые
кэши хранят в `text` уже ОЗВУЧЕННЫЙ текст; переиспользовать его как `rawText` нельзя
(`render` отработает поверх раскрытого → неверные позиции ударений). Поэтому Фаза 0
**один раз инвалидирует кэш** через bump `version` в `SentenceCacheEntry`. Это
разовая пере-обработка книг; зато ПОСЛЕ неё улучшения лингвистики синтеза (Фаза 1)
кэш не трогают вовсе. Дальше `version` нужен только при смене схемы очистки/
токенизации (они меняют `rawText`), что редко.

## 6. Русское качество (за `RussianProfile`)

Это и есть «максимум на русском». По убыванию слышимого эффекта:

1. **Ударения** (`RussianStress`) — ✅ механизм реализован (S4). Профиль возвращает
   **позиции** ударений в `SpokenMarkup`; `SileroBackend` рендерит `+` после ударной
   гласной (спайк §9), `AVSpeechBackend` отбрасывает. Seed-словарь из 20 однозначно
   часто-неверно-произносимых слов (звони́т, догово́р, катало́г…). Проверено
   end-to-end: stressed-аудио Silero ≠ plain.
   **S4b (бэклог) — контекстная дизамбигуация омографов** (за́мок/замо́к, му́ка/мука́):
   плоский словарь их не решает, нужен анализ окружения (предлоги, согласование,
   POS). Расширять словарь однозначными словами — безопасно и инкрементально.
2. **Согласование числительных** (`RussianNumerals`). Сейчас только именительный
   («на 2 страницах» → «на два страницах»). Поэтапно: (а) сохранить текущее как
   дефолт, (б) добавить согласование для частых конструкций «предлог + число + сущ.».
3. **Аббревиатуры/единицы** (`RussianAbbreviations`). Расширить словарь, разрешать
   «г.» (год/город/грамм), «в.» (век/в.), «см» (сантиметр/смотри) по контексту.

## 7. Дорожная карта (фазы)

| Фаза | Содержание | Зачем |
|------|-----------|-------|
| **0a. Русский путь** | кэш хранит `rawText` + `render` на воспроизведении (разовый bump `version`); `Sentence.language`; разнести `TextNormalizer` → `RussianProfile` + `TextPipeline`; **извлечь `SpeechBackend`** (спайк §9 → Silero основной движок для ударений) | Открывает дешёвую Фазу 1 + делает Silero первоклассным |
| **1. Русская лингвистика** | `RussianStress` (рендер `+` в `SileroBackend`), согласование числительных, словарь аббревиатур | Потолок качества на русском |
| **0b. Робастность загрузки** | постраничный `classifyPage` (`.text/.ocr/.skip`) + единый конвейер очистки + отдельный OCR-lane | Смешанные PDF + паритет OCR |
| **2. Робастность макетов** | `ColumnOrderer` (колонки), `ContentFilter` (код/формулы/таблицы) | Учебники, статьи |
| **3. Будущее** | `EnglishProfile`, детект языка → голос, `CloudBackend` | Мультиязычность, новые голоса |

**Порядок намеренно не по номерам, а по цели «русский в первую очередь»:**
0a → 1 (русское качество) → 0b (робастность типов) → 2 → 3. Фаза 0a — единственный
рефакторинг, обязательный для дешёвой Фазы 1; 0b откладывается до работы со сканами/
смешанными, если только спайк ударений не потребует сделать Silero основным движком
раньше (тогда извлечение `SpeechBackend` из 0b поднимается вперёд).

Корректностный фикс `requestPriorityLoad` для OCR-документов
(`ReaderViewModel.swift:320`) — дёшев, привязан к 0b (трогает загрузку OCR).

**Спайки (§9) выполняются ДО всего** — их результат может изменить приоритет
движков и форму API ударений.

---

## 8. Что НЕ строим сейчас (YAGNI-границы)

- Полный мультиязычный фреймворк (локали, RTL, переговоры о языке) — до реального
  второго языка.
- Детект языка контента — отложено по решению (сейчас только русский).
- `LanguageProfile` НЕ должен превратиться в god-протокол: только то, что доказанно
  меняется от языка к языку. Если для одного языка — оставляем в `RussianProfile`
  напрямую, протокол извлекаем при появлении второго.
- Облачный TTS — только шов `SpeechBackend`, без реализации.
- **Классификация страниц и порядок чтения — функции, НЕ протоколы** (§3.1, §3.5).
  Реализаций по одной; протокол под одну реализацию — спекуляция. Извлекаем протокол
  только когда появляется реальная вторая стратегия.

`+` после ударной гласной (НЕ `U+0301`). AVSpeech лексическое ударение омографов не
поддерживает. Следствия:
- Для «максимума на русском» **Silero — основной движок**, AVSpeech — офлайн-фолбэк.
- Извлечение `SpeechBackend` (было в Фазе 0b) **поднимается вперёд** — оно нужно, чтобы

- `AVSpeechBackend` просто **отбрасывает** `SpokenMarkup.stresses` (graceful degradation —
  архитектура это уже допускает); `SileroBackend` рендерит их как `+`.
- ⚠️ Тест на super-compact Milena (macOS, q=1). Перед финальной фиксацией —
